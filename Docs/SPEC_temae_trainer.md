# 要件定義・仕様書:裏千家 薄茶平点前 MRトレーニングアプリ TEMAE://PROTOCOL(点前プロトコル)

- 対象デバイス: Meta Quest 3S(パススルーMR)
- 開発環境: Unity + Meta XR SDK、Claude Code による生成AI支援開発
- 開発形態: 個人・プライベート開発(軽量ボリューム最優先)
- 版: v0.2(部屋条件・風炉・操作系の決定を反映)

---

## 1. 目的と背景

裏千家茶道の薄茶・平点前(運び点前)の稽古を、自宅で一人でも反復できるようにする。
MR(パススルー)を用いて、現実の畳(または床)の上に以下を重畳表示する。

1. 足運びガイド: 左足・右足をどの順で、畳のどこに置くかを現実空間にマッピング
2. 手順ナビ: 「今やる所作」と「次の所作」を提示
3. (発展)道具ガイド: 道具の置き位置・扱い方のヒント表示

### ゴール(このプロジェクトで達成したいこと)
- 平点前の「順番を覚える」段階の学習を、本や動画より直感的にする
- 週末の細切れ時間で開発を継続できる規模に収める

### ノンゴール(やらないこと)
- 所作の正誤判定・採点(足や道具の実トラッキングは Quest では困難)
- 畳の自動画像認識(後述の通り手動キャリブレーションで代替)
- マルチユーザー、ネットワーク機能、点前の網羅(薄茶平点前のみ)

---

## 2. 前提となる技術的制約(重要)

要件を左右する Quest 3S の制約を先に整理する。

| 制約 | 内容 | 本アプリでの対応方針 |
|---|---|---|
| 足のトラッキング不可 | Quest は頭と手(+コントローラ)のみ追跡。足の位置は取得できない | 足跡ガイドは「表示のみ」。実際に正しく踏めたかの判定はしない。ステップ送りはユーザー操作(ピンチ等)で行う |
| 畳の自動認識は非現実的 | パススルー映像へのアプリアクセスは制限があり、畳の縁を画像認識するのは開発負荷が高い | 起動時に畳の角をユーザーが指定する手動キャリブレーション方式を採用 |
| 道具の実物認識不可 | 実物の棗・茶杓等の位置は取得できない | 道具は「置き位置のゴースト(半透明モデル)」として仮想表示するに留める |
| 正座・低視点での利用 | 点前は座って行う。ガーディアン境界やトラッキングは床付近でも概ね動作するが、UI は低い視点前提で設計する必要がある | UI パネルは点前座の正面やや上に固定配置 |

この整理により、アプリの性格は「AR定規付きの手順書」であり、「審判」ではない、と定義する。

## 3. 想定利用シーン・ユーザー

- ユーザー: 裏千家の稽古経験者(開発者本人)。用語(茶道口、点前座、居前、建付など)は既知とする
- 場所A: 実際の畳のある和室(メイン想定)。自宅環境は江戸間8畳だが、**アプリは部屋全体をモデル化せず、点前で実際に使用する畳(点前畳+茶道口からの経路上の畳、計2〜3枚)のみをキャリブレーションして扱う**(§6.3)。これにより部屋の畳割りの事前定義が不要になり、別の部屋・稽古場でもそのまま使える
- 点前の季節: **風炉** の薄茶平点前を収録対象とする(炉は将来拡張)
- 場所B: 洋室・フローリング(補助想定)→ 仮想畳モードで畳グリッド自体を床に描画して代替
- 利用の流れ: 装着 → キャリブレーション → 足運び練習 or 通し手順練習 → 終了

仮想畳モードは「畳がない環境でも使える」だけでなく、キャリブレーション処理を共通化できるため、実装コストがほぼ増えない(畳の四隅指定 → グリッド生成のロジックは同一)。

## 4. 機能要件(フェーズ分割)

継続性を最優先し、各フェーズ単体で「動いて嬉しい」状態になるよう分割する。

### Phase 0: 開発環境立ち上げ(機能なし)
- Unity プロジェクト作成、Meta XR SDK 導入、Quest 3S 実機でパススルー表示
- ハンドトラッキングでピンチ操作を検出できる
- 完了条件: 実機でパススルー上に立方体1個が置け、ピンチで色が変わる

### Phase 1: 畳キャリブレーション + 足運びガイド(MVP)
- F1-1 床平面の取得(Meta の Scene 設定 or 平面検出)
- F1-2 使用畳キャリブレーション: まず点前畳の四隅をピンチで指定して基準座標系を確立。続いて経路上の畳(1〜2枚)は、基準畳からのグリッド外挿でプレビュー表示し、実際の縁とずれていれば縁上の点を1点タップして補正(向き90度切替もここで指定)。部屋全体は扱わない
- F1-3 使用畳の描画: キャリブレーションした畳のみ輪郭+縁を薄く描画(OFF可)。仮想畳モードでは同じ2〜3枚ぶんの畳を床平面上の任意位置に配置して代替
- F1-4 足跡ガイド表示: 左足=青・右足=赤などで足跡マークを畳上に表示。番号と順序アニメーション(次に踏む足が点滅/浮遊)付き
- F1-5 ステップ送り: §7.1 の操作系(注視ドウェルを主、ピンチを副)で「次へ / 戻る / 最初から」
- F1-6 キャリブレーションの永続化(Phase 1.5): Meta の Spatial Anchor(空間アンカー)で畳座標系の原点を保存し、次回起動時に復元を試みる。復元に失敗した場合・ずれていると感じた場合は 1 操作で再キャリブレーションに落とせる導線を必ず用意する。キャリブレーション結果を「原点 Transform 1 個」に集約する設計にしておけば、永続化の追加は小さな差分で済む
- 収録シーケンス: 入室〜点前座に着くまで、および退出の足運び(平点前の歩行部分)
- 完了条件: 和室で装着し、入室の足運びを最後まで順に表示できる

### Phase 2: 手順ナビ(点前の進行ガイド)
- F2-1 手順データの定義: 薄茶平点前の全所作をステップ列として JSON で定義(§6)
- F2-2 手順パネル UI: 「現在の所作(大きく)」「次の所作(小さく)」を表示。任意で要点メモ(例: 「茶杓は二引きで清める」)を1行添える
- F2-3 進行操作: Phase 1 と同じ操作系で共通化(§7.1)。ハンズフリー送りが成立する前提で、**データ粒度は「1ステップ=1所作」の細かい単位で確定**。UI 側はセクション見出しで「まとまり」も同時に見せる(細かいデータは粗く束ねられるが、逆は作り直しになるため)
- F2-4 セクションジャンプ: 「道具運び出し」「茶を点てる」「仕舞い」などの章単位で頭出し
- F2-5 音声読み上げ: スコープ外(決定)。手順データに音声用フィールドは設けず、必要になった時点でスキーマ拡張する
- 完了条件: 通しで手順を最初から最後まで追える

### Phase 3: 道具配置ガイド(発展)
- F3-1 道具ゴースト表示: 棗・茶碗・茶杓・茶筅・建水・柄杓(風炉/炉は当面片方に限定)の置き位置を半透明モデルで畳座標系上に表示
- F3-2 手順連動: 現在ステップに関係する道具ゴーストをハイライト
- F3-3 (任意)動きのヒント: 「帛紗さばき」等の複雑所作は 3D 化せず、パネル内の短い動画/連続画像で示す(3D アニメ制作は重いので原則回避)
- 完了条件: 手順進行に応じて道具の置き位置が切り替わって見える

### 将来アイデア(スコープ外・メモのみ)
- ハンドトラッキングによる簡易ジェスチャ確認(帛紗を持つ手の左右チェック程度)
- 炉/風炉の切り替え、他の点前への拡張、稽古ログ

## 5. 非機能要件

- パフォーマンス: 72fps 維持。パススルー + 半透明オブジェクト少数なので余裕がある想定。ポストプロセス・リアルタイム影は使わない
- オフライン動作: ネットワーク不要。データはアプリ内蔵 JSON
- 起動から練習開始まで: キャリブレーション済みなら 30 秒以内(アンカーの永続化で再キャリブレーション省略、Phase 1 では毎回キャリブレーションでも可)
- 開発ボリューム: 各フェーズが週末数回(目安 10〜20 時間)で完了する粒度に収める。超えそうな機能は「任意」へ降格する
- コンテンツの正確性: 足運び・手順の内容は開発者自身が教本・稽古経験に基づき監修する(アプリはデータを表示するだけの構造にし、内容修正が JSON 編集で済むようにする)

## 6. データ設計(手順・足運びの表現)

手順・足運び・道具配置をすべてデータ駆動にし、コードと点前知識を分離する。Claude Code での開発とも相性が良い(コード生成と内容監修を分けられる)。

### 6.1 座標系
- 原点: 基準畳(点前畳)の茶道口側・下座角(要検討)
- 単位: メートル。畳サイズはプリセット(京間 1.91×0.955m、江戸間 1.76×0.88m など)で正規化座標 → 実座標に変換
- 足跡は「畳ローカル正規化座標(0..1) + 向き角度」で記述し、畳サイズが変わっても破綻しないようにする

### 6.2 JSON スキーマ案(抜粋)

```json
{
  "temae": "usucha_hiradate",
  "school": "urasenke",
  "sections": [
    {
      "id": "nyushitsu",
      "label": "入室",
      "steps": [
        {
          "id": "s001",
          "text": "茶道口で一礼",
          "note": "襖の開閉は省略(MVP)",
          "footsteps": [],
          "tools": []
        },
        {
          "id": "s002",
          "text": "左足から入り、点前畳へ進む",
          "footsteps": [
            { "foot": "L", "tatami": "approach1", "u": 0.15, "v": 0.5, "yawDeg": 0, "order": 1 },
            { "foot": "R", "tatami": "approach1", "u": 0.35, "v": 0.5, "yawDeg": 0, "order": 2 }
          ],
          "tools": [
            { "id": "chawan", "state": "carried" }
          ]
        }
      ]
    }
  ]
}
```

- `footsteps` が空のステップは手順ナビのみ表示
- `tools` は Phase 3 まで無視して良い(スキーマだけ先に切っておく)

### 6.3 使用畳モデル(部屋全体はモデル化しない)

足運びデータは部屋の畳番号ではなく、**畳の「役割」** で参照する。

| 役割ID | 意味 | キャリブレーション方法 |
|---|---|---|
| `temae` | 点前畳(基準座標系) | 四隅をピンチ指定(必須) |
| `approach1`, `approach2` | 茶道口から点前畳への経路上の畳 | グリッド外挿プレビュー → ずれていれば縁1点タップで補正、向きは90度切替で指定 |

```json
{
  "calibration": {
    "tatamiSize": { "length": 1.74, "width": 0.87 },
    "requiredTatami": ["temae", "approach1"]
  }
}
```

- `tatamiSize` は江戸間の公称値を初期値とし、`temae` の四隅実測で上書き補正。`approach` 側の外挿にも補正後の寸法を使う
- この方式の利点: 部屋の畳割り・写真・事前定義が一切不要。自宅8畳でも稽古場でも、使用畳の位置関係が同じ点前(運びの平点前)ならそのまま動く
- 制約: 経路が長い点前(広間の遠い茶道口など)に拡張する場合は `approach` を増やすだけで対応可能
- 注意: 上記の足位置・歩数は仮の値。実際の裏千家の足運び(踏み込む足、畳の縁を踏まない、一畳の歩数、回り方 など)は監修時に確定させる。この仕様書は正とせず、データ入力時に教本と照合すること

## 7. UX / 画面フロー

```
[起動] → パススルー開始
   ↓
[モード選択] 実畳モード / 仮想畳モード、畳サイズプリセット選択
   ↓
[キャリブレーション] 畳の角をピンチで指定 → グリッドのプレビュー → 確定
   ↓
[メニュー] ① 足運び練習  ② 通し手順  ③ 設定(表示ON/OFF等)
   ↓
[練習中]
   - 床: 畳グリッド(薄表示・OFF可) + 足跡ガイド + 道具ゴースト(Phase3)
   - 空中パネル: 現在/次の所作、セクション名、進捗
   - 操作: §7.1 参照(注視ドウェル主体+ピンチ併用)
```

### 7.1 操作系の設計(ハンズフリー要件)

点前中は両手が塞がる(茶碗・帛紗・柄杓を持つ、両手で扱う)場面が多く、ピンチ単独は操作系として不成立と判断。以下の3系統を併用する。

| 方式 | 内容 | 用途 | 実装コスト |
|---|---|---|---|
| ① 注視ドウェル(主) | パネル脇に「次へ」ボタンを常設し、頭の向き(ヘッドゲイズ)でレイを飛ばして約1秒見続けると発火。発火まで円形プログレスを表示し誤操作を防ぐ | 点前中の標準操作。完全ハンズフリー | 小(Quest 3S に視線トラッキングはないが、頭の向きだけで十分機能する。レイキャスト+タイマーのみ) |
| ② ピンチ(副) | 右手ピンチ=次へ、左手ピンチ=戻る | 手が空いている場面(足運び練習、メニュー操作)での素早い操作 | 極小(SDK 標準) |
| ③ オートプレイ(補) | 一定間隔(可変・例 5〜15 秒)で自動送り。ドウェルボタンで一時停止/再開 | 通し稽古で「テンポに乗る」用途。操作自体を無くす選択肢 | 小 |

- 「戻る」はドウェルでも可能にする(「次へ」の反対側に小さめ配置。誤発火防止のためドウェル時間を長めに)
- 音声認識は魅力的だが、Meta Voice SDK の導入・ネットワーク依存・認識調整の負荷が本プロジェクトの軽量方針に合わないためスコープ外。将来検討するなら、認識を伴わない「マイク音量しきい値トリガー(『はい』等の発声で次へ)」が最軽量の代替候補としてメモしておく
- 設計上の含意: ①が成立するため、手順粒度は「1所作=1ステップ」の細かさでも進行の手間にならない(§F2-3 の決定根拠)

- 文字サイズ・パネル距離は正座視点(目線高さ約 0.9〜1.1m)で読める前提で調整。ドウェルボタンは正座時の視界内・かつ普段の所作で見ない方向(パネル脇)に置き、誤発火を避ける
- 色覚多様性に配慮し、左右の足跡は色+形(L/R 文字、形状差)で区別

## 8. 技術スタック・構成

| 項目 | 選定 | 理由 |
|---|---|---|
| Unity バージョン | Unity 6(6000.x LTS)または 2022.3 LTS | Meta XR SDK の対応が厚い。2018 からの再入門なら最新 LTS で始めるのが結局早い |
| レンダリング | URP | Quest 標準。Built-in は避ける |
| XR SDK | Meta XR All-in-One SDK(Building Blocks 利用)+ MRUK | パススルー・ハンドトラッキング・平面検出をブロック追加でほぼノーコード導入できる |
| 入力 | ハンドトラッキング主体(コントローラはフォールバック) | 点前の練習中に手が塞がるのは不自然なため |
| データ | StreamingAssets 内 JSON | 内容修正をビルド不要に近づける(エディタ再生で確認) |
| 開発補助 | Meta Quest Developer Hub、Quest Link(PC 直結での再生確認) | ビルド往復を減らしイテレーションを速くする |

### プロジェクト構成案

```
Assets/
  App/
    Scripts/
      Calibration/   // 畳座標系の確立・保存
      Guide/         // 足跡・道具ゴーストの生成と進行制御
      Sequence/      // JSONロード、ステートマシン(現在ステップ管理)
      UI/            // 手順パネル、メニュー
    Data/            // temae JSON
    Prefabs/         // 足跡、畳グリッド、パネル
```

コアは「Sequence(ステートマシン)が現在ステップを持ち、Guide と UI がそれを購読して表示を更新する」だけの単純な一方向データフローとする。

## 9. Claude Code での開発方針

- リポジトリ直下に `CLAUDE.md` を置き、本仕様書・座標系の定義・プロジェクト構成・「Meta XR の Building Blocks で生成された部分は手で触らない」等の約束事を記載する
- Unity は GUI 操作(シーン配置、SDK 設定)と切り離せないため、役割分担を明確にする:
  - Claude Code 向き: C# スクリプト一式、JSON スキーマとバリデータ、エディタ拡張(足跡位置をシーンビューで編集して JSON に書き出すツール等)
  - 人間がやる: パッケージ導入、Building Blocks 追加、ビルド設定、実機確認
- 足跡データの入力が最も面倒な作業になる見込みのため、「エディタ上で畳を上から見てクリックで足跡を置く簡易オーサリングツール」を Phase 1 の後半で Claude Code に作らせると、以降の内容作成が大幅に楽になる

## 10. マイルストーンとリスク

### マイルストーン(目安)
1. M0: 実機パススルー Hello World(Phase 0)
2. M1: 入室の足運びが和室で表示できる(Phase 1)← 最初の「見せられる」成果
3. M2: 薄茶平点前を通しで手順ナビできる(Phase 2)
4. M3: 主要道具のゴースト表示(Phase 3)

### リスクと対応
| リスク | 対応 |
|---|---|
| Unity/Quest 再入門の初速が出ない | Phase 0 を独立マイルストーン化し、公式チュートリアル(パススルー導入)をなぞるだけで完了とする |
| キャリブレーション精度が悪く足跡がずれる | 四隅指定 → 最小二乗で矩形フィット。ずれても「順番の学習」という目的は損なわれないと割り切る |
| 手順データ入力が苦行化して頓挫 | §9 のオーサリングツールを先に作る。まず「入室のみ」など最小データで通す |
| 点前内容の誤り | データ駆動にして JSON 修正のみで直せる構造にする。教本と照合してから他人に見せる |
| スコープ膨張(採点したくなる等) | ノンゴール(§1)に明記した項目は次プロジェクト扱いにする |

## 11. 決定事項と残課題

### 決定事項(v0.2)
1. 部屋: 部屋全体はモデル化せず、使用する畳(点前畳+経路畳2〜3枚)のみをキャリブレーションする方式(§6.3)。自宅は江戸間8畳だが、アプリはこれに依存しない
2. 点前: 風炉の薄茶平点前
